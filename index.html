Module 1

AI 基礎與資訊搜尋

AI 的知識從哪來？如何讓它找到最正確的資訊？
Lesson 01

AI 新手 vs AI 進階用戶
AI Novice vs AI Power User

本節概念：Context · Neutral Questioning · Iterative Writing · Rubric

核心心智模型

把 AI 想成一個剛入職的聰明大學生：高度積極、知識廣博，但對你一無所知。你提供的 context 越好，他做的越好。

五個關鍵差異

① 問題的複雜度

❌ 新手

把 AI 當搜尋引擎，問簡單問題：
"Does Taco Bell still have the double deck taco?"

✅ 進階用戶

上傳大量文件（車子規格、保險方案、報價），讓 AI 花數分鐘深度分析後給出完整報告，附帶取捨比較。

② 提供的 Context 多寡

❌ 新手

給一個短 prompt，期望 AI 自己填空。
例："Please write a good self-review to send to my boss."

✅ 進階用戶

上傳專案追蹤截圖、最近的文件、語音備忘錄，再叫 AI 根據這些寫自我考核。AI 才能真正抓到你最自豪的成就。

③ 獲得誠實的回饋

❌ 新手（有偏誤的提問）

"我有個很棒的商業點子：流動紮染服務。幫我批評一下。"
→ AI 會說「真棒！」

✅ 進階用戶（使用 rubric）

提供評分標準：「用以下標準評估：有沒有市場需求？有沒有競爭優勢？」
→ AI 回答："8 分（滿分 100），原因如下…"

④ 寫作工作流程

❌ 新手

"幫我寫一篇關於黑莓機的部落格"
→ 直接得到 AI 味重、缺乏實質的文章（AI Slop）

✅ 進階用戶
  1. 先要求 AI 列出大綱
  2. 反覆批評、修改大綱
  3. 大綱確定後展開成條列式重點
  4. 最後才生成完整文章

⑤ 對 AI 犯錯的認知

重要認知
AI 犯錯的頻率被高估了。"草莓有幾個 R" 這類病毒式失敗案例不能代表 AI 的整體能力。現代 AI 能做：深度研究報告、個人健康數據分析、建立網站。

本節重點

  1. Context 越豐富，AI 的回答越精準。不要期望 AI 會猜你的意思。
  2. 提問時要中立，不要透露你期望的答案。
  3. 寫作要分階段進行，不要一次叫 AI 輸出最終成品。
  4. 提供 rubric 可以強迫 AI 給出更客觀的評估。
Lesson 02

AI 的知識來源
Pre-trained Knowledge

本節概念：Pre-training · Knowledge Cutoff · Training Data Frequency

AI 怎麼「學習」

AI 從「讀」大量網路文字中學習，這些學來的知識叫 pre-trained knowledge（預訓練知識）。

來源類型範例
社群媒體Reddit、Quora
書籍與百科Wikipedia
新聞網站各大媒體
學術論文各領域研究
合計「數兆個字」

訓練資料密度決定可靠程度

主題網路資料量AI 可靠度
料理、電影、名人極多高
常見事實（最高建築）多高
Quasar（天文專有名詞）少中等
你公司的內部文件零無法回答

語言比例的影響

大多數訓練資料是英文。廣東話使用者超過 8,000 萬人，但廣東話網路內容可能不到 0.1%，因此 AI 對小語種的掌握相對較弱。

對打字錯誤的容忍度

實用提示
AI 能理解大量錯字（因為訓練資料本身就有錯字）。打字時不必太在意小錯誤，直接送出即可，不需要為了修正錯字而浪費時間。

Pre-trained Knowledge 的限制

  • 含有誤解和過時資訊（網路上的錯誤資訊也被學進去了）
  • 有 Knowledge Cutoff Date（訓練截止日期），之後的事情一無所知
  • 某個日期後的事件，需要 web search 才能補充

本節重點

  1. 先估計「這個主題在網路上有多少資料」，就能預測 AI 的可靠度。
  2. 需要即時資訊、或問截止日期後的事，就要用 web search。
  3. 公司內部資料 AI 不會知道，必須透過 context 提供給它。
Lesson 03

Web Search：即時資訊的獲取
Web Search

本節概念：Knowledge Cutoff · Web Search Trigger · Source Reliability · Two-AI Architecture

什麼時候 AI 會觸發 Web Search？

提問情境是否用 Web Search原因
"I dropped my phone in soup. What to do?"否常識，不需即時資訊
"What's the 67 meme from 2025?"是截止日期後出現的事件
"Find me a highly rated gym near Mountain View"是即時性 + 地點特定
"What's on Voyager 1's vinyl record?"否歷史事實，不需更新

觸發 Web Search 的兩種方式

  1. AI 自己判斷要不要搜
  2. 在 prompt 中明確要求："請先做 web search 再回答"

Web Search 的底層架構（Two-AI Model）

底層原理

背後有兩個 AI 模型在合作：

  1. User-facing AI（你對話的那個）：負責最終回答
  2. Assistant AI：負責搜尋、過濾、摘要網頁

重要：User-facing AI 只看到摘要，沒有讀完整文章。這就是為什麼 AI 有時會誤解它引用的來源。

Web Search 的三大限制（重要！）

① 傾向引用社群媒體

根據研究報告，AI 最常引用的來源：第一名 Reddit、第二名 Wikipedia、第三名 YouTube…。社群媒體的可靠度參差不齊。

② 可能引用過時網頁

真實案例
Andrew Ng 用 AI 找跑步地點，AI 引用了一個 20 年前的網頁，推薦的學校早已不對外開放。

③ 來源引用可能失真

AI 只看到摘要，有時會聲稱某篇文章支持某結論，但實際去查原文卻完全不是那樣。

如何讓 AI 使用更可靠的來源

Prompt 範例
"請只使用官方健康機構（WHO、FDA、歐洲藥品管理局）的資訊來回答這個問題，避免使用 Reddit 或 blog。"

Web Search vs 直接用搜尋引擎

需求適合工具
快速找特定網站、掃瞄多個來源搜尋引擎（Google / Bing）
想要多個來源的綜合結論、有取捨的判斷AI + Web Search

本節重點

  1. 需要即時或地點特定資訊時，確保 web search 有開啟。
  2. 在 prompt 中指定信任的來源類型，AI 就會優先找更可靠的文章。
  3. AI 引用的文章不一定真的支持它說的結論，重要決策要去驗證原文。
Lesson 04

Deep Research：深度多來源研究
Deep Research

本節概念：Deep Research Mode · Research Plan · Multi-source Synthesis

三種資訊獲取方式完整對比

方式來源數量回應時間適用情境
Pre-trained Knowledge—秒常識問題、不需即時資訊
Web Search少數幾個數秒~數十秒即時資訊、地點特定查詢
Deep Research數十個以上數分鐘以上複雜問題、需要整合多元觀點

使用 Deep Research 的正確流程

  1. 給足夠的 context：例如「我在 Palo Alto，前院 30×20 英呎，預算 $500，想辦萬聖節鬼屋」
  2. AI 會先提出 Research Plan（研究計畫）
  3. 用戶審閱計畫，可以修改或補充
  4. 確認後讓 AI 執行：自動搜尋 → 讀取 → 整合 → 輸出報告
Andrew Ng 的評語
"Deep Research 是嚴重被低估的工具，很多人有這個功能卻沒有善用它。"

何時選擇 Deep Research

問題類型選擇
找一個事實（"最高建築是什麼"）Pre-trained / Web Search
今天天氣怎樣Web Search
"每日步數對長期健康的影響（需要看科學文獻）"Deep Research
"天氣如何影響杜拜旅遊業（需要整合多個面向）"Deep Research

本節重點

  1. 遇到「需要整合多個來源、有很多面向要考慮」的問題，主動切換 Deep Research 模式。
  2. 給 AI 的 context 越豐富，研究計畫越精準。
  3. 可以在執行前審核研究計畫，避免 AI 找錯方向。
Lesson 05

Context 管理：讓 AI 更懂你的任務
Context Management

本節概念：Context Window · System Prompt · Desktop Apps · Agentic Context

什麼是 Context？

Context 是 AI 用來生成回應的所有資訊，包含：

  • System prompt（AI 的基本指示：日期、名稱、能力、工具說明）
  • 你的 prompt
  • 對話歷史（你和 AI 的所有來回）
  • 你上傳的文件

Context 容量對比

容量
人類工作記憶約 7 件事
現代 AI（2026）約 750,000 字（≈ 前 4-5 本哈利波特）
思考要提供什麼 context 的方法：想像一個受信任的顧問需要知道什麼，才能給你真正有幫助的回答？把那些都提供給 AI。

何時要開新對話

常見錯誤

在討論自己的健身計畫到一半，突然問「幫我的媽媽設計健身計畫」。

→ 你自己的偏好和條件會汙染給媽媽的建議。開新對話可以清空 context，避免干擾。

AI Desktop Apps：代理式 Context 讀取

工具代表：Claude Code/Cowork、Microsoft Copilot、Google Gemini

核心能力：AI 可以自己去電腦裡找相關檔案，不需要你手動上傳。你只需要說「幫我整理這個資料夾」，AI 會自己瀏覽、讀取、重新命名、移動檔案。

使用 Desktop App 的安全注意事項

重要安全提醒
  1. 給 AI 存取最小必要資料夾，不要讓它讀整台電腦
  2. 仔細檢查每一個「權限請求」
  3. 被刪除的檔案不會進回收桶，無法還原
  4. 被編輯的檔案通常沒有編輯歷史，無法回復
  5. 建議流程：讓 AI 先提案 → 你審核 → 確認後再執行

本節重點

  1. Context 越相關，回答越精準。不相關的 context 反而會干擾。
  2. 切換全新主題時開新對話，管理 context 的純粹度。
  3. 使用 Desktop App 時要小心授權範圍，刪了就找不回來。
Lesson 06

Reasoning Models：讓 AI 深度思考
Reasoning & Thinking Models

本節概念：Reasoning · Thinking Mode · Ultra Think · Agentic Loop

AI 能完成任務的複雜度演進

時期AI 能完成的任務難度
早期（2022-2023）幾秒鐘能完成的簡單任務
2024-2025幾秒到幾分鐘的任務
現在（2026）人類需要數小時的任務

Reasoning 的運作邏輯（Agentic Loop）

運作流程

你的 prompt → AI 開始思考 → [判斷是否需要更多資訊] → 使用工具（web search / 讀檔案）→ 繼續思考 → [重複幾次] → 輸出最終答案

如何觸發更深度的 Reasoning

方法做法
介面觸發在 ChatGPT / Claude / Gemini 介面選擇「thinking mode」選項
Prompt 觸發在 prompt 中說 "think really hard" 或 "ultra think"

已過時的建議

舊建議（現已不適用）

"Tell the AI to think step by step"

→ 這在 2022-2023 年有效，現在的 AI 已能自行決定最佳推理方式，只需說 "think hard" 即可。

使用 Reasoning 的最佳實踐

  1. 用最新的模型：推理能力通常比 6-12 個月前的版本強非常多
  2. 給足 context：把一個人類專家需要知道的所有資訊都提供給 AI
  3. 給難題：不要只測試簡單問題，要嘗試真實的工作任務
  4. 說 "think hard" 或選 thinking mode：讓 AI 知道你願意等它多想幾分鐘

本節重點

  1. 複雜任務要明確觸發 reasoning，不要省略。
  2. Reasoning 可以花幾分鐘甚至更長，這是正常的，給它時間。
  3. 「think step by step」已過時，改用「think hard」。

Lesson 07

Brainstorming：腦力激盪的正確姿勢
Brainstorming with AI

本節概念：Iterative Brainstorming · Context-driven Creativity · Feedback Loop

關鍵統計

根據 OpenAI 分析 ChatGPT 對話的數據：創意發想（creative ideation）佔所有對話的 3.9%，寫作和實用指導佔約 50%。

為什麼 AI 的預設回答是「常識」？

AI 從網路學習，網路上寫「啞鈴彎舉」的文章遠比寫「貓觸發式微運動」多。所以它天生傾向給出「平均答案」——常見、通用、無趣。

要讓 AI 給出有創意的回答，必須透過「給足 Context」和「多輪迭代反饋」把它推離常識區。

兩種層次的 Brainstorming

層次做法結果
基本問「給我一個計畫的想法」常識性、通用的答案（深蹲、伏地挺身）
進階給足 context + 多輪迭代反饋更有創意、更個人化（蹦床休息、貓觸發訓練）

多輪迭代腦力激盪流程（Andrew Ng 推薦）

  1. 給 AI 基本 context，要求「給我 3-5 個選項」
  2. AI 給出選項（例：Plan A 保流動性、Plan B 先還最高利率、Plan C 先還家人借款）
  3. 給反饋：「我不喜歡 A 太被動，喜歡 B 的思路，順帶一提我剛收到 450 美元現金而且快搬家了」
  4. AI 根據反饋生成新的 3-5 個選項
  5. 重複直到找到滿意的方案，再叫 AI 展開細節
關鍵洞見
反饋不只是挑選，它同時也是在告訴 AI「哪種 context 對你的問題更重要」。這是找出正確 context 最有效率的方式。

本節重點

  1. 不要只叫 AI 生成一個答案，要叫它給幾個選項。
  2. 迭代給反饋，每一輪反饋都在幫 AI 更了解你的需求。
  3. 給越多個人化 context（設備、限制、偏好），越容易得到有創意的回答。
Lesson 08

Sycophancy：避免 AI 只說你想聽的話
Avoiding Sycophancy

本節概念：Sycophancy · Neutral Framing · Rubric · Training Feedback Loop

Sycophancy 是什麼？

Sycophancy（諂媚）：AI 被訓練成「讓用戶滿意」，所以天生傾向告訴你它認為你想聽的話，而非客觀的事實。

Sycophancy 的成因（訓練循環）

  1. 用戶問：「我覺得當內向者比較好，你不也這麼認為嗎？」
  2. AI 若回答「確實有很多優點…」→ 用戶點讚 ✅
  3. AI 若回答「不一定，各有優缺點」→ 用戶不開心 ❌
  4. AI 學到：「同意用戶 = 獲得正向回饋」→ 越來越傾向同意
研究數據
根據 Washington Post 的研究，ChatGPT 同意用戶的頻率 vs 不同意的頻率 = 約 10:1。某些模型甚至說過："你剛才說的東西很深刻，你 100% 正確。"

如何識別 Sycophantic Prompt

❌ 會引發諂媚的提問
  • "Don't you think remote work is better?"（已暗示立場）
  • "Is it true that office work is more productive?"（已暗示立場）
  • "Find all the positive measures this quarter"（只找正面）
  • "Aren't carbon taxes bad for small businesses?"（已暗示答案）
✅ 中立的提問方式
  • "What are the pros and cons of remote vs in-office work?"
  • "To what extent do carbon taxes affect small businesses?"
  • "What does current research say about AI's effect on jobs?"
  • "How does productivity compare between remote and in-office work?"

Rubric 的力量：強制客觀評估

提供清楚的評分標準，讓 AI 無法猜到你期望的答案方向：

Rubric 範例（商業點子評估）
請用以下標準評估這個商業想法（100分）：
- 市場需求是否存在（25分）：是/否
- 有沒有目標市場（25分）：是/否
- 是否有競爭優勢（25分）：是/否
- 執行可行性（25分）：是/否

請逐項評分後加總，最後說明得分低的原因。
好的 Rubric 每一項標準都是客觀的、是/否題，沒有灰色地帶。這樣 AI 就無法主觀判斷。

本節重點

  1. 中立提問是最重要的反 sycophancy 技巧：不要在問題中透露你期望的答案。
  2. 需要客觀評估時，提供 rubric 可以有效抑制 AI 的諂媚傾向。
  3. 如果不確定怎麼設計 rubric，可以先讓 AI 幫你設計，再用來評估。
Lesson 09

Writing：用 AI 寫出有深度的文章
AI Writing & Progressive Outlining

本節概念：AI Slop · Progressive Outlining · Iterative Drafting

關鍵統計

根據 OpenAI 的數據，寫作任務佔 ChatGPT 所有請求的 24%，是單一最大類別。

什麼是 AI Slop（AI 廢文）？

AI Slop 是看起來寫得很好、但讀起來毫無實質內容的 AI 生成文字。

AI Slop 的特徵說明
大量 em dash（—）ChatGPT 發布後在社群媒體明顯增加
過度使用 "nuanced"、"delve"AI 特有的高頻詞
愛用「三件事並列」比真人更頻繁的結構模式
"Not X but Y" 的曖昧說法"It's not about infrastructure, it's about architecture"
大量形容詞、缺少具體名詞"robust and highly insightful"
社會現象
根據調查，40% 的美國雇員在過去一個月內收到過工作 AI slop。更有趣的是，人類自己也開始在演講和 podcast 中更常用 "delve" 這個詞——AI 的語言習慣正在影響人類。

Progressive Outlining：避免 AI Slop 的核心方法

❌ 錯誤做法

"Write a blog post about BlackBerry phones"

→ 直接得到 AI slop

✅ 正確流程
  1. 研究：讓 AI 搜尋支持/反對論點的證據
  2. 大綱選項：叫 AI 給 3 個不同大綱
  3. 反饋大綱：修改方向、調整結構
  4. 迭代大綱：直到滿意
  5. 展開重點：先展開成條列式
  6. 最終文章：最後才生成完整文字

為什麼先做大綱比直接寫文章更有效率？

槓桿原理

改一個大綱的標題 → 整個對應段落都會改（高槓桿）

改文章裡的某個句子 → 只改了那一句（低槓桿）

→ 在大綱階段修改的投資報酬率遠高於在最終文章階段修改。

本節重點

  1. 不要直接叫 AI 生成最終文章，先做大綱。
  2. 大綱和重點迭代到滿意之後，再生成完整文字。
  3. AI 是「思考夥伴」，不是「一次性輸出機器」。
Lesson 10

Critiquing & Editing：用 AI 精煉你的作品
AI Editing & Cross-model Review

本節概念：Incremental Editing · Rubric-based Critique · Jagged Intelligence · Cross-model Review

關鍵統計

寫作任務中，約 2/3 是修改既有文章，而非從零開始創作。

分段編輯（逐句或逐段）

不要一次貼整篇文章讓 AI 改，而是一次只改一句或一段：

  1. 貼上一個句子
  2. 請 AI 提供幾種改寫方式（punchy 版、visionary 版、conversational 版…）
  3. 挑選喜歡的或繼續迭代
  4. 確定後進到下一段

好處：聚焦、可控、不會讓大量改動同時出現讓你看不清楚什麼改了。

Rubric 評分：取得整體客觀評估

❌ 差的 Rubric（會引發諂媚）

"請評估這篇短篇科幻故事，並給一個 100 分制的分數。"

→ AI 會先在心中決定分數，再想辦法合理化。結果幾乎都是高分。

✅ 好的 Rubric（客觀是/否標準）
項目配分標準（是/否）
角色25每個有名字的角色都有明確目標
情節25有清楚的衝突和解決方式
世界建構25有至少 2 個獨特的世界細節
寫作技巧25段落之間有轉場句
實用技巧
不知道怎麼設計 rubric？先讓 AI 幫你設計一份，再用來評估你的作品。

Cross-model Review（跨模型審閱）

讓 ChatGPT 生成內容，再讓 Gemini（或 Claude）用 rubric 評審。整合兩個模型的知識，效果略優於讓同一個模型審閱自己的輸出。

AI 的「鋸齒形智能」（Jagged Intelligence）

AI 比人類強的地方AI 比人類弱的地方
快速閱讀大量網頁某些需要身體感知的任務
解複雜數學題需要最新即時知識的任務
同時處理大量文件高度創意、需要真實生命經驗的創作
建議習慣
定期把同一個 prompt 丟給不同 AI（ChatGPT、Claude、Gemini），維持對各模型能力的直覺。AI 市場競爭激烈，各模型的優缺點每幾個月就會大幅改變。

本節重點

  1. 編輯要分段進行，一次專注於一個部分。
  2. 整體評估要用客觀 rubric，避免 sycophancy 影響分數。
  3. 偶爾嘗試 cross-model review，讓不同模型互相補充。

Multimodal AI：多模態輸入輸出
Multimodal Inputs & Outputs

本節概念：Modality · Input/Output Cost · Ethics of Generative AI

輸出類型成本對比

輸出類型速度成本備註
文字最快最低AI 的核心強項，LLM 從語言起步
語音中等中低近年品質大幅提升，已接近真人
圖片數十秒中高一次生成整張，不能中途停止
影片最慢最高比圖片貴非常多

重要趨勢

成本趨勢
生成任何格式的成本都在快速下降。一年後，生成圖片、影片的成本都會比現在更低。

AI 生成內容的倫理面

應用有益有害
語音生成Podcast 後製修音、電動遊戲角色配音假冒家人聲音進行詐騙
圖像生成設計師原型、個人創作深度偽造、不實形象
影片生成低成本特效製作假新聞影片

本節重點

  1. 多模態應用最重要的限制是成本和速度：圖片和影片的迭代能力遠低於文字。
  2. 之前學的 prompting 技巧（給 context、多選項）也適用於多模態，但需要更多等待時間。
  3. 有益應用遠多於有害應用，但要有意識地只用於負責任的目的。
Lesson 12

圖像理解：讓 AI 看懂你的圖片
Image Understanding

本節概念：Visual Understanding · Coarse Vision · Image as Context

AI 圖像理解的能力與限制

情境AI 表現原因
讀取白板上的字（即使被遮擋）不錯，能推斷能從周圍脈絡推理
辨識視覺上很獨特的物品不錯特徵明顯，即使模糊也能辨識
區分外觀相似的健身器材差各器材看起來太像，細節容易錯
讀取收據、帳單金額中等能做，但值得複查
辨識手寫文字中等能做，但值得複查
AI 傾向「粗略地看圖」，可能錯過細節。視覺上越獨特的東西，AI 越容易正確辨識。

實際應用場景

  • 拍下白板、便利貼照片 → 要 AI 整理今天腦力激盪的討論摘要
  • 上傳手寫文字 → 要 AI 轉錄成打字版
  • 上傳收據 → 要 AI 計算你應該付多少
  • 上傳圖片 → 問 AI「你會怎麼描述這張圖？」→ 學習圖像描述語言，幫助之後的圖像生成
實用技巧
一張圖片抵過千字解釋。當你很難用文字描述你想要的東西時，直接上傳圖片讓 AI 看。

本節重點

  1. 重要資訊（金額、數字）要複查，不能完全信任 AI 的圖像閱讀。
  2. 視覺獨特 = AI 容易識別；視覺相似 = AI 容易搞混。
  3. 可以同時上傳多張圖片（如腦力激盪的多張白板照）讓 AI 整合。
Lesson 13

圖像生成：Diffusion Model 的邏輯
Image Generation & Diffusion Models

本節概念：Diffusion Model · Art Vocabulary · Image Prompt Writing

Diffusion Model 的運作原理

底層原理：擴散模型

訓練時：AI 看大量「圖片 + 描述文字」，學習從「純噪音（隨機像素）」→ 逐步去噪 → 得到清晰圖片。

生成時：你輸入 prompt，AI 從純噪音開始，逐步「去噪」，一次生成整張完整圖片（不是逐行、逐像素）。

圖像生成的已知缺陷（早期版本）

  • 手的外觀異常（超過或少於 5 根手指）
  • 文字扭曲（生日快樂 → 亂碼）
  • 同一角色在不同畫格外觀不一致

現代模型（2026 年）已大幅改善，但仍非完美。

如何提升圖像生成品質

① 學習圖像描述語言

英文詞彙效果
cinematic電影感光影
watercolor水彩風格
cyberpunk賽博龐克
anime動漫風
photorealistic真實照片感

熟悉藝術語彙的人天生擅長圖像生成。可以上傳圖片問 AI「你會怎麼描述這張圖？」來學習正確的描述詞彙。

② 讓文字 AI 幫你寫圖像 prompt

流程範例
  1. 告訴文字 AI（Claude/ChatGPT）："幫我寫一個圖像生成 prompt：一隻貓在夜晚偷偷經營咖啡廳"
  2. 文字 AI 會生成包含場景、角色細節、氛圍/風格的完整描述
  3. 把這個描述丟給圖像 AI（Midjourney/Stable Diffusion/Nano Banana）

圖像生成 vs 文字生成的迭代差異

文字生成圖像生成
速度秒數十秒以上
成本最低中高
可以中途停止？是否（一次生成整張）
迭代容易度容易大量迭代受限，要等待

本節重點

  1. 學習視覺藝術的描述語言，是提升圖像生成品質最有效的投資。
  2. 不知道怎麼寫圖像 prompt？先讓文字 AI 幫你寫。
  3. 圖像生成的隨機性是特性不是缺點，多生幾次挑選最喜歡的。
Lesson 14

用 AI 建 App：從想法到程式
Building Apps with AI

本節概念：Goal-Input-Output Framework · Simple vs Complex Apps

建 App Prompt 的三要素

框架：Goal → Input → Output
目標（Goal）：我要做什麼？
輸入（Input）：用戶要輸入什麼？
輸出（Output）：App 要顯示什麼？

範例：
"Build a fun fireworks simulator.
Input: user clicks on the screen.
Output: colorful fireworks display."

適合新手嘗試的簡單 App（三個特徵）

  1. 有一個明確、單一的功能
  2. 不需要上傳外部檔案或串接外部 API
  3. 使用一段時間後可以關掉

簡單 vs 困難的判斷標準

簡單（適合新手）困難（需要有開發背景）
單人遊戲多人連線遊戲
法文單字練習（閃卡）即時 AI 回饋的對話練習
番茄鐘計時器需要串接即時 API 的工具
分帳計算機有複雜狀態管理的應用
靜態色彩工具跨裝置同步功能
建議策略
不確定難不難？就直接試。最壞的結果是它做不好，你從失敗中學到這類任務的邊界在哪裡。

本節重點

  1. 先用 Goal-Input-Output 框架設計 prompt，讓 AI 清楚知道要做什麼。
  2. 從最簡單的 App 開始，單一功能、無外部依賴。
  3. 可以把建好的 App 分享給朋友用，大多數工具都有分享連結功能。
Lesson 15

資料分析：讓 AI 幫你寫 Code 跑分析
Data Analysis with AI Code Execution

本節概念：Code Execution Tool · Data Insights · Agentic Analysis

AI 資料分析的能力

你只需要上傳資料和說明需求，AI 會自己寫 Python 程式碼並執行，生成圖表和洞見。

支援的資料類型可以產出什麼
個人健康數據（心率、跑步時間）趨勢圖、進步分析
銷售報表（電子表格）銷售趨勢圖、關鍵指標
任何表格式數據統計摘要、比較圖

AI Code Execution 的底層邏輯

運作流程

你的 prompt + 數據文件 → AI 判斷需要計算 → AI 寫 Python 程式碼 → 呼叫「執行程式碼」工具 → 取得計算結果 → 生成圖表或文字摘要給你

何時 AI 選擇用 Code？

情境AI 的選擇
常識問題使用 pre-trained knowledge
即時 / 地點特定資訊使用 web search
複雜多面向研究使用 deep research
需要計算、繪圖、草稿寫並執行 code

實例：珍珠奶茶店銷售分析

Prompt 範例

"哪些飲料的銷售變化最大？幫我畫圖。"（附上 Excel 銷售資料）

→ AI 識別出：草莓抹茶春天爆紅、椰奶茶秋天亮眼

→ 生成具有清晰顏色編碼的折線圖

進階版："Create a one-slide year-in-review graphic. Analyze the data carefully for insights."（加 "carefully" 觸發深度思考）

重要提醒
AI 用 code 算出的數字通常比純文字回答更可靠（有實際執行結果）。但仍建議：核對關鍵數字是否和你的預期吻合，避免因為資料格式問題導致計算錯誤。

本節重點

  1. 有表格數據要分析，就直接上傳給 AI，說明你想知道什麼，讓它寫 code 跑。
  2. 在 prompt 中加 "carefully" 可以觸發 AI 更深入的分析思考。
  3. AI 資料分析比不上頂尖人類數據科學家，但對基礎分析非常有效率。
